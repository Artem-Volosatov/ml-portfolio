## ML Проекты

### Классификатор научных статей arXiv

* **Ссылка на папку проекта:** [`./ML_projects/arxiv-classifier/`](./ML_projects/arxiv-classifier/)
* **Живое демо:** [HF Spaces](https://huggingface.co/spaces/artem-volosatov/arxiv-classifier)

Мультилейбл-классификатор научных статей по ~130 категориям arXiv на дообученном **SciBERT**. По заголовку и аннотации возвращает все подходящие рубрики с вероятностями (у статьи их обычно несколько, поэтому задача поставлена как мультилейбл, а не мультикласс). Модель дообучена на ~163k статьях ([датасет TimSchopf/arxiv_categories](https://huggingface.co/datasets/TimSchopf/arxiv_categories)) на A100 за 10 эпох с bf16 и batch 128, лосс — binary cross-entropy. Классы сильно несбалансированы, поэтому качество мерилось по micro/macro-F1; отдельным шагом подбирался порог отсечения (снижение с 0.5 до 0.25 дало бесплатный прирост обеих метрик). Итог — Micro-F1 0.71 / Macro-F1 0.55. Демо собрано на **Streamlit** и развёрнуто в **Docker** на HF Spaces, с готовыми примерами из свежих статей. Подробный разбор обучения, кривые и подбор порога — в [отчёте](./ML_projects/arxiv-classifier/report.pdf).

**Ключевые техники:** PyTorch, 🤗 Transformers, SciBERT, Streamlit, Docker, multi-label classification.

### Web-сервер для обучения и использования ML-моделей

* **Ссылка на папку проекта:** [`./ML_projects/ML_Server/`](./ML_projects/ML_Server/)

Многозадачный REST API сервис для обучения, инференса и управления моделями машинного обучения, реализованный на **FastAPI** с контейнеризацией в **Docker**. Сервер поддерживает параллельное обучение нескольких моделей с контролем ресурсов (ограничение по числу ядер CPU и максимальному количеству одновременно загруженных моделей для инференса), реализует полный жизненный цикл ML-моделей через API-методы: обучение (`fit`), предсказание (`predict`), загрузка/выгрузка в память (`load`/`unload`), удаление с диска (`remove`/`remove_all`). Поддерживается три дискриминативных алгоритма классификации/регрессии. Проект включает клиентскую часть с демонстрацией синхронных запросов через **requests** и асинхронных через **aiohttp**, с корректной обработкой граничных случаев и ошибок сервера. Конфигурация сервиса осуществляется через переменные окружения (`.env`), развёртывание — в 1-2 команды через Docker.

**Ключевые техники:** FastAPI, Docker, multiprocessing, requests, aiohttp, scikit-learn.

### Бот для поиска фильмов

* **Ссылка на папку проекта:** [`Other_Projects/cinemabot/`](./Other_Projects/cinemabot/)
* **Бот в Telegram:** [@YetAnotherCineBot](https://t.me/YetAnotherCineBot)

Telegram-бот для поиска информации о фильмах и сериалах, написанный на **aiogram 3.x** в полностью асинхронном стиле. Пользователь отправляет название фильма текстом, бот ищет его через **Kinopoisk Unofficial API** и возвращает карточку с постером, рейтингом, описанием, режиссёром и актёрским составом, а также инлайн-кнопки с ссылками для просмотра. Поддерживается история поисков и статистика (`/history`, `/stats`), которые хранятся в **SQLite** через асинхронный драйвер **aiosqlite**. Архитектура разделена на слои: хендлеры команд, сервисы (клиент Kinopoisk поверх переиспользуемой `aiohttp`-сессии), слой работы с БД и утилиты форматирования. Логика форматирования и парсинга API покрыта юнит-тестами (**pytest**). Конфигурация — через переменные окружения (`.env`), развёртывание — одной командой через **Docker Compose**; бот работает в режиме long-polling, что не требует открытых портов и публичного домена.

Сейчас бот задеплоен на личном VPS сервере вместе с другой нагрузкой типичной для личного VPS сервера. Использован бесплатный API с ограниченным числа запросов, поэтому бот работает в качестве демо.

**Ключевые техники:** aiogram, aiohttp, aiosqlite, Docker Compose, pytest, async/await.

## Контесты

### House Prices
Контест проводился в ноябре 2025г среди студентов ШАД.
Цель: предсказать цену недвижимости в Сиэтле. Метрика MAPE
Ограничение: не использовать нейросетевые методы.
Метод решения: градиентный бустинг (CatBoostRegressor)
Результат: 0,1090 MAPE. 9 место из 112 участников

Главный инстайт из соревнования - переход к более удобному таргету позволил мгновенно получить хороший результат. Смена таргета с цены на цену за фут улучшил MAPE с 32% до 16%. Переход от цены за фут к логарифму цены за фут улучшил MAPE с 16% до 11.5%. Оставшася часть решения - генерация признаков.

### Autonomy Transport BEV
Контест проводилс в мае 2026 среди студентов ШАД.
Цель: по 4 фотографиям с камер беспилотного автомобиля построить BEW-карту с препятствиями. Метрика IoU
Метод решения: Unet + геометрические соображения
Результат: IoU=0.5391, 22 место из 276 участников

В этом контесте достатоно хороший бэйзлайн получался просто усреднением всего трейна (потому что дороги, типичная ситуация - свободно прямо, препятствия на границе дороги). И модели быстро сваливались в переобучение после 1-2 эпох.

В итоговом решении я делал проекцию изображения камер на плоскость. Это давало изображение, в котором геометрия совпадает с целевой картинкой. Также я пробовал делать одновременно несколько проекций на разных высотах, это в итоге дало небольшой, но значимый прирост. Очень важным оказался подбор порога.

Моей главной ошибкой было то, что я пробовал в качестве энкодера в юнете только разные версии ResNET. Победитель контеста поэкспериментировал с разными энкодерами и DINOv2 тут начинала учиться и давала хороший результат.

### Orcs
Контест проводился в декабре 2025г среди студентов AI Masters
Цель: даны 2 крайне зашумленных списка людей. Нужно найти в списке сотрудников (2 млн) тех, кто присутствует списке орков (50к). Метрика IoU.
Ограничение: метрические методы
Метод решение: очистка от опечаток, проверка всех подозрительных через кастомную метрику сходства
Результат: IoU=0.9396, 7 место из 96 участников

В этой задаче у меня получился пайплайн из 3 ноутбуков. Первые два чистят данные от опечаток. В первом все данные приводятся к одному формату (символы, кейс и т.п.). Во втором проходимся по именам сверху вниз по популярности - если имя близко (по левинштейну) к многократно более популярному - считаем опечаткой. Так удалось значимо вычистить ФИО.

В последнем файле происходит непосредственно анализ. Написал функцию, которая по паре людей считает скор их похожести (совпал инн +3 балла, отличая в 2х цифрах - +1 балл, не совпал пол - -1 балл и т.п.). Дальше я находил по разным критериям (например, Fuzzy поиск по инн с расстоянием 2 или полное совпадение имя + отчество) пары кандидатов и вычислял для этих кандидатов скор.

В итоговом решении было много параметров, которые тюнились вручную методом "смотрим глазами на пары с пограничным скором"

### Ozon Fresh

Цель: по истории покупок предсказать, совершит ли пользователь любую покупку за следующий месяц, метрика ROC AUC

Результат: ROC AUC = 0.82183


Решение, в основном, состояло из экспериментов по генерации признаков, прогона катбуста и замеров их влияния на результат.
